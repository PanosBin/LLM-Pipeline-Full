# Survey Cases Experiment: LLM Vulnerability Triage

## Overview

This experiment evaluates whether **LLM-generated context summaries** improve the accuracy of automated vulnerability triage. It uses the **same 10 vulnerability cases** presented to human participants in the survey, allowing direct comparison between human and LLM performance.

The experiment mirrors the survey's between-subjects design:
- **Baseline (Control Group)**: LLM receives only SAST tool information
- **Enhanced (Experimental Group)**: LLM receives SAST tool information + pre-generated method/class/cluster summaries

## Ground Truth

The 10 cases were manually classified by expert review. The ground truth is **never revealed to the LLM** during prediction -- it is only used for scoring after the experiment.

| Case | File | Vulnerability Type | Ground Truth | Reasoning |
|------|------|--------------------|--------------|-----------|
| 1 | GetTransactions.java (DVB) | Hidden UI Element | **False Positive** | `setVisibility(View.GONE)` is a normal Android UI pattern for hiding/showing views |
| 2 | SendMoney.java (DVB) | Sensitive Data in Log | **True Positive** | `Log.d()` outputs decrypted bank transaction response data |
| 3 | LogActivity.java (DIVA) | Sensitive Data in Log | **True Positive** | `Log.e()` logs credit card information entered by user |
| 4 | AccessControl3NotesActivity.java (DIVA) | Hidden UI Element | **False Positive** | `setVisibility(View.INVISIBLE)` is a normal Android UI pattern for PIN fields |
| 5 | SecureURLOpen.java (DodoBank) | SSL Certificate Bypass | **False Positive** | Demo/test SSL code with empty `checkServerTrusted()`, not production bypass |
| 6 | TransactionTable.java (DodoBank) | SQLite Raw Query Injection | **False Positive** | SQL query uses internal `id` parameter, not user-controlled input |
| 7 | WeakCrypto.java (OVAA) | AES ECB Mode | **True Positive** | `Cipher.getInstance("AES")` defaults to insecure ECB mode |
| 8 | Encryption.java (PIVAA) | Static IV in AES-CBC | **True Positive** | Hardcoded all-zero IV `{0,0,0,...}` makes CBC mode predictable |
| 9 | Authentication.java (PIVAA) | Sensitive Data in Log | **True Positive** | `Log.i()` logs plaintext username and password |
| 10 | Digest.java (PIVAA) | Sensitive Data in Log | **False Positive** | Apache Commons Codec library code, not application-level logging |

**Distribution**: 5 True Positives, 5 False Positives (balanced)

## Experiment Design

### Phase 1: Baseline Predictions (Control)

The LLM receives **only SAST tool information** (identical to what the survey control group saw):
- Vulnerability Type
- CWE identifier
- Severity level
- OWASP Mobile category
- Description
- File name
- Flagged line number
- Flagged code snippet

The LLM must classify (TRUE_POSITIVE / FALSE_POSITIVE) and rate confidence (1-5).

### Phase 2: Enhanced Predictions (Experimental)

The LLM receives **SAST tool information + context summaries** (identical to what the survey experimental group saw):
- All fields from Phase 1
- Method Summary
- Class Summary
- Cluster Summary

The LLM must classify (TRUE_POSITIVE / FALSE_POSITIVE) and rate confidence (1-5).

### Phase 3: Summary Quality Evaluation

The LLM evaluates the quality of the summaries themselves:
- **Helpful**: Did summaries provide useful context for triage?
- **Accuracy**: Are the summaries factually accurate about the code?
- **Relevance**: Do summaries focus on security-relevant aspects?
- **Completeness**: How complete are the summaries? (1-5)
- **Missing Info**: What critical information is missing?
- **Improvement**: How could summaries be improved?

### Confidence Scale (matches survey)

| Score | Label |
|-------|-------|
| 1 | Not at all confident |
| 2 | Slightly confident |
| 3 | Moderately confident |
| 4 | Very confident |
| 5 | Extremely confident |

## Prompt Engineering

### Key Design Decisions

1. **False Positive Awareness**: The prompts explicitly inform the LLM that SAST tools have high false positive rates (40-60%), preventing the bias of classifying everything as TRUE_POSITIVE.

2. **Critical Evaluation Criteria**: The prompts instruct the LLM to consider:
   - Is the flagged pattern actually dangerous or a common safe Android pattern?
   - Does the code actually handle sensitive data or is it benign?
   - Is this library/framework code vs application code?
   - Could this be test/demo code rather than production code?

3. **No Ground Truth Leakage**: The ground truth classification is never included in any prompt sent to the LLM. It is only used for post-hoc accuracy scoring.

4. **CodeLlama Instruct Format**: All prompts use the `[INST] ... [/INST]` format required by CodeLlama-7b-Instruct-hf.

5. **Structured Output**: Prompts enforce exact response format (`CLASSIFICATION:`, `CONFIDENCE:`, `REASONING:`) for reliable parsing.

### Generation Parameters

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Model | CodeLlama-7b-Instruct-hf | Same model used in the pipeline for consistency |
| Temperature | 0.3 | Low enough for consistency, high enough to avoid deterministic bias |
| top_p | 0.9 | Standard nucleus sampling |
| max_new_tokens | 500 | Sufficient for structured response |
| Quantization | 8-bit (bitsandbytes) | Fits in GPU memory |

## Running the Experiment

### Prerequisites

- Server with setup completed (`bash setup_server.sh`)
- GPU with CUDA support (tested on NVIDIA A100 80GB)
- HuggingFace token configured in `.env`

### Execute

```bash
/root/.local/bin/poetry run python experiment_survey_cases.py
```

**Runtime**: ~10-15 minutes on A100 GPU (20 predictions + 10 evaluations)

### Output

All results are saved to `experiment_survey_YYYYMMDD_HHMMSS/`:

| File | Description |
|------|-------------|
| `experiment_results.json` | Complete raw results with all predictions, confidence scores, reasoning, and raw LLM responses |
| `summaries_used.json` | The pre-generated summaries used in the Enhanced condition |
| `analysis.json` | Computed metrics: accuracy, precision, recall, F1, confidence means, per-case breakdowns |
| `tables.tex` | Ready-to-use LaTeX tables for the thesis/paper |
| `fig1_metrics_comparison.pdf/png` | Grouped bar chart: Baseline vs Enhanced (Accuracy, Precision, Recall, F1) |
| `fig2_per_case_accuracy.pdf/png` | Per-case accuracy comparison with ground truth labels |
| `fig3_confidence_distribution.pdf/png` | Confidence histograms on 1-5 scale for both conditions |
| `fig4_confusion_matrices.pdf/png` | Aggregated confusion matrices for both conditions |
| `fig5_accuracy_stability.pdf/png` | Accuracy across runs (if NUM_RUNS > 1) |
| `fig6_per_case_confidence.pdf/png` | Per-case average confidence comparison |
| `fig7_summary_quality.pdf/png` | Summary quality assessment (Helpful/Accurate/Relevant percentages) |

## Metrics Computed

| Metric | Description |
|--------|-------------|
| **Accuracy** | Correct predictions / total cases |
| **Precision** | True positives correctly predicted / all predicted as TP |
| **Recall** | True positives correctly predicted / all actual TPs |
| **F1 Score** | Harmonic mean of precision and recall |
| **TP Detection** | Accuracy on true positive cases only |
| **FP Detection** | Accuracy on false positive cases only (critical for triage) |
| **Confidence** | Average confidence score (1-5 scale) |

## Comparison with Survey Results

The experiment enables direct comparison between:

| Participant | Baseline (SAST only) | Enhanced (SAST + Summaries) |
|------------|---------------------|----------------------------|
| **Human (Survey)** | Control group results | Experimental group results |
| **LLM (This Experiment)** | Phase 1 results | Phase 2 results |

Key comparisons for the thesis:
1. Does the LLM benefit from summaries the same way humans do?
2. Is the LLM better/worse than humans at detecting false positives?
3. Does confidence correlate with accuracy for both humans and LLM?
4. Which specific cases are hardest for humans vs LLM?
